# TIP-006: Declarative Straight-Line Code - Reducing Control-Flow Crystallization Radius

**Status:** Recommended for AI-generated and AI-maintained codebases.

## Context

AI agents struggle when business logic is hidden inside procedural control flow.

Nested conditions are only one part of the problem.

The larger issue is manual execution management:

* branching;
* loops;
* retries;
* locks;
* transactions;
* resource scopes;
* exception tunnels;
* callback chains;
* lifecycle handling.

Humans can compress these execution structures into mental models.

AI agents must repeatedly reconstruct them from source code.

Every explicit control-flow construct increases the amount of execution state the agent must track before making a safe change.

We need Declarative Straight-Line Code.

---

## The Problem

Many codebases express business behavior through low-level procedural mechanics:

```pseudo
try {
    lock resource {
        for item in items {
            if item is valid {
                with transaction {
                    result = process(item)

                    if result failed {
                        retry result
                    }

                    if result succeeded {
                        save result
                    }
                }
            }
        }
    }
}
```

The actual business intent is small.

But the agent must reason about:

* lock lifetime;
* transaction scope;
* loop state;
* retry behavior;
* exception propagation;
* conditional execution;
* partial failure states.

The agent spends context on execution mechanics instead of business meaning.

---

## Declarative Straight-Line Code

Declarative Straight-Line Code moves execution mechanics into trusted abstractions, libraries, language constructs, or prepared domain classes.

Instead of forcing the agent to manually write procedural control flow, the code expresses intent directly:

```pseudo
valid_items = items.filter(is_valid)

results = valid_items
    .process_with(retry_policy)
    .inside(transaction_boundary)
    .with_concurrency_limit(1)

results.save_successful()
```

The business operation becomes visible.

The execution policy is explicit.

The procedural machinery is no longer scattered through the method body.

---

## The Principle

Do not make the agent manually manage execution state when the platform, library, or prepared class can express it declaratively.

Prefer:

```pseudo
orders.process_all_with(policy)
```

over:

```pseudo
for order in orders {
    try {
        if order is valid {
            process(order)
        }
    } catch error {
        retry_or_log(error)
    }
}
```

The goal is not fewer lines.

The goal is less execution-state tracking.

---

## What Should Be Made Declarative

AI Oriented Architecture should push recurring control-flow mechanics into explicit reusable constructs:

* validation pipelines instead of nested checks;
* collection operators instead of manual loops;
* retry policies instead of hand-written retry loops;
* transaction wrappers instead of inline transaction plumbing;
* resource scopes instead of manual open/close handling;
* concurrency primitives instead of manual locks;
* result types instead of exception tunnels;
* event handlers instead of synchronous orchestration chains.

The agent should compose prepared execution primitives, not rebuild them repeatedly.

---

## Prepared Classes

Declarative Straight-Line Code often requires preparing the codebase for AI agents.

This means creating classes, helpers, or framework-level constructs that collapse procedural mechanics into safe, explicit operations.

Examples:

```pseudo
RetryPolicy.standard().execute(command)
```

```pseudo
TransactionBoundary.run(operation)
```

```pseudo
ValidatedCollection.from(items).process(processor)
```

```pseudo
EventFlow.emit(command).then_wait_for(result_event)
```

These constructs reduce control-flow complexity at the point of business modification.

The agent no longer writes retry loops, lock scopes, or transaction scaffolding manually.

It composes known primitives.

---

## Guard Clauses Still Matter

Guard Clauses remain useful, but they are only one tactic.

They reduce branching by exiting early:

```pseudo
if request is missing:
    fail InvalidRequest

if account is inactive:
    fail InactiveAccount

withdraw(account, amount)
```

But Guard Clauses solve only conditional nesting.

Declarative Straight-Line Code targets the broader problem:

manual procedural control flow.

---

## Why It Matters

Declarative Straight-Line Code improves:

* retrieval precision;
* patch predictability;
* generation stability;
* test isolation;
* execution-path visibility;
* local reasoning.

Most importantly, it reduces the amount of procedural context an AI agent must reconstruct before changing business behavior.

The agent sees:

```pseudo
apply_discount(order, discount_policy)
```

instead of:

```pseudo
loop → validate → lock → transact → try → retry → branch → save
```

That difference matters.

---

## Relationship to AI Oriented Architecture

AI Oriented Architecture optimizes systems for Local Reasoning.

Declarative Straight-Line Code reduces Control-Flow Crystallization Radius.

A component may have good boundaries and still be difficult for an agent to modify if its internal execution path is procedural, nested, and state-heavy.

The solution is not to remove all control flow.

The solution is to move recurring control-flow mechanics into stable, explicit constructs.

Business code should describe what happens.

Prepared infrastructure should define how execution is controlled.

---

## What This Is Not

Declarative Straight-Line Code does not mean:

* hiding important business rules;
* eliminating all loops;
* eliminating all error handling;
* replacing clear code with magical frameworks.

It means avoiding manual procedural machinery in business methods when a stable declarative primitive would express the same intent with lower reasoning overhead.

If an abstraction hides business meaning, it increases Crystallization Radius.

If it hides repetitive execution machinery, it reduces it.

---

## Recommendations

For AI engineering teams:

* Replace manual loops with collection/pipeline operations when practical.
* Replace hand-written retry logic with explicit retry policies.
* Replace inline transaction plumbing with transaction boundaries.
* Replace manual locks with safe concurrency primitives.
* Replace exception tunnels with explicit result types where appropriate.
* Keep Guard Clauses for visible business preconditions.
* Prepare reusable execution primitives before asking agents to modify complex flows.
* Measure success by reduced Control-Flow Crystallization Radius.

Stop forcing AI agents to manually reconstruct procedural machinery.

Make execution policy explicit.

Make business intent linear.
