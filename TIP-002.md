### TIP-002: Semantic Collision - The Erasure of Domain Boundaries

Status: Already happening.

### Context

As AI agents move from helper scripts into core business logic, a new failure mode appears:

Semantic Collision.

When multiple domain concepts are linguistically similar, the model starts treating them as interchangeable.

### What Happened

A code generation workflow contained three separate concepts:

* Trading Day Time
* Tuning Day Time
* Eval Day Time

The generated code silently collapsed all three into a single identifier.

The system remained:

* syntactically correct;
* type-safe;
* fully deployable;
* passing conventional tests.

But the business semantics were gone.

Trading logic was overwritten by evaluation logic.

### Why This Is Dangerous

Most teams now consume AI-generated code faster than they review it.

If the code compiles and tests are green, semantic corruption often goes unnoticed - especially in domains where reviewers lack deep business context.

We are starting to lose one of the most important safety layers in software engineering:

human domain intuition.

### Consequences

* Silent logic corruption
* Passing tests with incorrect behavior
* Loss of domain identity
* Extremely expensive debugging
* Gradual semantic drift across bounded contexts

### Recommendations

For architects:

* Stop exposing primitive types in domain-critical flows
* Use semantic value objects
* Treat naming as architecture, not cosmetics
* Add semantic validation to AI-generated code
* Strictly separate bounded contexts

The compiler protects syntax.

It does not protect meaning.
