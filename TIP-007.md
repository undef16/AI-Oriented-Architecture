# TIP-007: Strict JSON Gateways and Auditable DTOs - Stop Defensive Programming in Business Logic

**Status:** Recommended for cleaner business logic, safer AI-generated code, and explainable runtime state.

## Context

AI coding agents become overly defensive when input contracts are unclear.

If a method receives loose JSON, partially validated objects, nullable fields, or ambiguous runtime structures, the agent starts generating protection code everywhere:

* null checks;
* fallback values;
* type checks;
* defensive parsing;
* deep optional-field handling;
* speculative error branches.

This creates a dangerous pattern.

Business logic becomes mixed with input sanitation.

The agent spends tokens protecting the code from invalid states instead of implementing business behavior.

We need deterministic contracts at the boundary.

We also need explainable state inside the system.

---

## The Problem

Many systems allow dynamic input uncertainty to leak into core handlers.

A business method receives an object and still has to ask:

* does this object exist?
* are required fields present?
* are field types valid?
* are enum values known?
* are nested objects populated?
* do defaults need to be reconstructed?

When those questions remain open, the AI agent fills the gap with defensive boilerplate.

The result is code that is longer, noisier, harder to review, and easier to corrupt during generation.

---

## Strict JSON Gateways

A Strict JSON Gateway is a hard validation boundary at the edge of a feature, service, or actor.

Its job is simple:

1. Accept loose external input.
2. Validate it against a strict schema.
3. Reject malformed payloads immediately.
4. Convert valid input into typed DTO contracts.
5. Pass only valid DTOs into business logic.

The core rule:

> Loose JSON must die at the boundary.

Once the request crosses the gateway, business code should operate on valid, typed, non-ambiguous contracts.

---

## Deterministic Contracts

In AI Oriented Architecture, DTOs are boundary contracts.

Their responsibility is simple:

* define required fields;
* define allowed types;
* define nullability rules;
* define structural guarantees;
* eliminate ambiguity at the system boundary.

A DTO answers:

> What data is allowed to enter this boundary?

Example:

```pseudo
contract PromoRequest {
    required code: NonEmptyString
    required customer_id: CustomerId
}
```

Once a request passes through a Strict JSON Gateway, business logic should be able to trust the contract.

Business methods should not repeatedly validate what the boundary has already guaranteed.

---

## Explainable State

Input contracts solve only half of the problem.

Even when data enters the system through deterministic DTOs, runtime state can still become difficult to understand.

Consider:

```pseudo
CustomerProfile {
    status = Suspended
}
```

The current state is visible.

The path that produced it is not.

To understand why the customer became suspended, an AI agent may need to inspect:

* logs;
* event streams;
* handlers;
* services;
* repositories;
* workflow history.

This increases reasoning cost and expands the amount of context required for debugging.

---

## Auditable DTOs

When runtime state provenance matters, DTOs can be extended into Auditable DTOs (ADTOs).

An ADTO preserves the normal DTO contract while additionally recording meaningful business state transitions.

Example:

```pseudo
CustomerProfile {
    status = Suspended

    history = [
        Created
        EmailVerified
        PaymentFailed
        Suspended
    ]
}
```

The object stores both state and provenance.

The objective is not auditing.

The objective is explainability.

An ADTO answers:

> How did this state become what it is?

---

## Defensive Bloat

Without a strict gateway, business logic often looks like this:

```pseudo
function apply_promo(request) {
    if request is null:
        return

    if request.code is missing:
        throw InvalidPromoCode

    code = trim(request.code)

    if code is empty:
        throw InvalidPromoCode

    if request.customer_id is missing:
        throw InvalidCustomer

    promo = database.find_promo(code)

    if promo is null:
        throw PromoNotFound

    apply(promo, request.customer_id)
}
```

Some of this is business logic.

Much of it is contract uncertainty.

The agent has to modify code inside a noisy execution path.

---

## Strict Contract Flow

With a strict gateway, malformed input is rejected before business execution begins:

```pseudo
gateway PromoRequestGateway {
    accepts json
    validates PromoRequestSchema
    returns PromoRequest
}
```

The business method receives a trusted contract:

```pseudo
function apply_promo(request: PromoRequest) {
    promo = database.find_promo(request.code)

    if promo not found:
        throw PromoNotFound

    apply(promo, request.customer_id)
}
```

The remaining validation is business validation, not technical sanitation.

That distinction matters.

---

## Perimeter Validation

Strict JSON Gateways push technical uncertainty to the perimeter.

Good candidates for perimeter validation:

* required fields;
* JSON shape;
* primitive types;
* enum membership;
* ID format;
* string normalization;
* basic range checks;
* schema version compatibility.

Good candidates for business logic:

* whether a promo exists;
* whether a customer is eligible;
* whether a discount can be combined;
* whether an account is allowed to perform an action;
* whether a workflow state transition is valid.

The gateway protects structure.

The business layer protects meaning.

---

## Mutation Slices

ADTOs should not record every setter invocation.

That creates noise.

Instead, they should capture meaningful business transitions.

Example:

```pseudo
ApplyPaymentFailurePolicy

status:
    Active -> Suspended

risk_score:
    20 -> 95
```

The emphasis is on business meaning rather than implementation details.

One business operation should typically produce one mutation slice.

Not one audit record per property assignment.

Example structure:

```pseudo
structure PropertyChange {
    property_name: String
    old_value: Any
    new_value: Any
}

structure TraceEntry {
    timestamp: Datetime
    operation: String
    caller_class: String
    caller_method: String
    reason: String
    changes: List[PropertyChange]
    snapshot: Object
}

abstract class ADTO {
    private history: List[TraceEntry] = []

    function record_mutation(operation, reason, changes) {
        history.add(
            TraceEntry(
                timestamp = Clock.utc_now(),
                operation = operation,
                caller_class = StackTrace.external_class(),
                caller_method = StackTrace.external_method(),
                reason = reason,
                changes = deep_clone(changes),
                snapshot = deep_clone(this)
            )
        )
    }

    function mutation_history(): List[TraceEntry] {
        return history
    }
}
```

The goal is explainability, not compliance auditing.

---

## State Provenance

Every significant mutation should answer:

* what changed;
* who changed it;
* why it changed;
* when it happened.

Example:

```pseudo
Mutation {
    operation: ApplyDiscount
    timestamp: 2026-05-01T10:15:00Z
    reason: LoyaltyCampaign

    changes:
        total:
            100 -> 80
}
```

Without ADTOs, debugging often looks like this:

```text
Current State
    -> Search Logs
    -> Search Events
    -> Search Handlers
    -> Search Repositories
    -> Guess Root Cause
```

With ADTOs:

```text
Current State
    -> Inspect Mutation History
    -> Identify Root Cause
```

The debugging path becomes shorter because the explanation travels with the object.

## Implementation Examples

Strict gateways can be implemented with:

* Zod or Valibot in TypeScript;
* Pydantic in Python;
* Records and validation attributes in C#;
* Java Records and Bean Validation;
* Go structs with strict decoding and validation tags;
* JSON Schema with generated typed contracts.

The tool does not matter.

The boundary does.

Auditable DTOs can be implemented with:

* mutation journals;
* embedded provenance metadata;
* event-derived state histories;
* lightweight state-transition records;
* domain-event aggregation.

The implementation is secondary.

The architectural goal is explainable state.

---

## Why It Matters

Strict JSON Gateways improve AI-generated code because they reduce ambiguity.

The agent no longer needs to guess whether every field may be missing, malformed, or null.

It can work against a stable contract.

This improves:

* business-logic density;
* patch predictability;
* test clarity;
* DTO consistency;
* agent reasoning;
* code review quality.

Most importantly:

the agent writes less defensive infrastructure and more domain behavior.

Auditable DTOs improve runtime reasoning.

They reduce the effort required to understand why important state transitions occurred.

Instead of reconstructing history from scattered infrastructure artifacts, agents can inspect provenance directly where the state exists.

---

## Relationship to AI Oriented Architecture

AI Oriented Architecture depends on local reasoning.

Strict JSON Gateways help by separating two concerns:

* input validity;
* business correctness.

Auditable DTOs extend that same principle to runtime state.

They separate:

* current state;
* state provenance.

When these concerns are mixed, every business method becomes harder for an agent to understand and modify.

When they are separated, the business path becomes smaller, cleaner, and safer to change.

When provenance travels with the object, agents do not need to reconstruct large portions of the system to understand why the object reached a particular state.

---

## What This Is Not

Strict JSON Gateways do not mean ignoring production reality.

Systems still need:

* observability;
* error reporting;
* schema evolution;
* backward compatibility;
* safe handling of untrusted clients.

The point is not to pretend invalid input cannot exist.

The point is to keep invalid input from leaking into core business execution.

Auditable DTOs do not mean:

* storing every setter call;
* creating massive object histories;
* replacing logs;
* replacing tracing systems;
* replacing event streams.

The objective is not perfect historical reconstruction.

The objective is making important state transitions self-explanatory.

---

## Recommendations

For AI engineering teams:

* Treat DTOs as boundary contracts, not loose data bags.
* Reject malformed JSON at the gateway.
* Use non-nullable required fields for mandatory parameters.
* Keep technical validation outside business handlers.
* Keep business validation inside business logic.
* Avoid speculative fallback chains in core methods.
* Generate typed contracts from schemas where possible.
* Make input guarantees explicit before asking agents to modify business flows.
* Use Auditable DTOs when state provenance matters.
* Record business-significant mutations instead of low-level setter noise.
* Capture why state changed, not just what changed.
* Expose mutation history to debugging tools and AI agents.

Stop forcing AI agents to write runtime shields inside your core code.

Lock down the entry contract at the boundary.

Let the agent work with valid business facts.

Stop forcing agents to reconstruct object history from scattered infrastructure artifacts.

Let important state explain itself.
