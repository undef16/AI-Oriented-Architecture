# TIP-009: Auditable DTOs (ADTO) — Making Runtime State Explainable

**Status:** Recommended for AI-assisted debugging, tracing, and state analysis.

## Context

Modern systems invest heavily in code observability.

We track:

* logs;
* metrics;
* traces;
* events;
* exceptions.

Yet one critical question often remains difficult to answer:

> How did this object reach its current state?

When an AI agent investigates a bug, the problem is rarely reading the current value.

The problem is reconstructing the journey.

The agent must discover:

* where a value originated;
* which component modified it;
* which business operation changed it;
* when the change occurred;
* why the change occurred.

This reconstruction process is expensive.

We need Auditable DTOs.

---

## The Problem

Traditional DTOs are excellent contracts.

They define:

* structure;
* types;
* required fields;
* serialization boundaries.

But they do not explain history.

Consider:

```pseudo
CustomerProfile {
    status = Suspended
}
```

The current value is visible.

The path that produced it is not.

To reconstruct the state transition, the agent may need to:

* inspect multiple handlers;
* search logs;
* follow event chains;
* traverse repositories;
* analyze execution history.

The object stores state.

It does not store provenance.

---

## DTO vs ADTO

In AI Oriented Architecture:

### DTO

A DTO is a contract.

Its responsibility is safe boundary communication.

```pseudo
CustomerProfileDto
```

It answers:

> What data exists?

---

### ADTO

An Auditable DTO is a runtime state carrier.

```pseudo
CustomerProfileAdto
```

It answers:

> How did the data become what it is?

An ADTO combines state with mutation history.

---

## Mutation History

An ADTO records significant state transitions.

Example:

```pseudo
CustomerProfile {
    status = Suspended

    history = [
        Created
        EmailVerified
        PaymentFailed
        Suspended
    ]
}
```

The current state remains visible.

The path to the state becomes visible as well.

---

## Mutation Slices

Tracking every field assignment is usually noise.

Instead, ADTOs focus on meaningful mutation slices.

Example:

```pseudo
ApplyPaymentFailurePolicy
    status:
        Active → Suspended

    risk_score:
        20 → 95
```

The emphasis is on business transitions rather than low-level setters.

This creates explanations that both humans and agents can understand.

```pseudo
structure PropertyChange {
    property_name: String
    new_value: Any
}

structure TraceEntry {
    timestamp: Datetime
    caller_class: String
    caller_method: String
    changes: List[PropertyChange]
    snapshot: Object
}

abstract class ADTO {
    private history: List[TraceEntry] = []
    private trace_enabled: Boolean = Config.get("ADTO_TRACE_ENABLED")

    constructor() {
        if trace_enabled {
            bind_property_triggers(this)
        }
    }

    private on_property_changed(property_name: String, new_value: Any) {
        if !trace_enabled {
            return
        }

        caller = StackTrace.get_external_caller()
        last = history.last_or_null()

        change = PropertyChange(
            property_name = property_name,
            new_value = deep_clone(new_value)
        )

        if last != null
           and last.caller_class == caller.class
           and last.caller_method == caller.method {

            last.changes.upsert_by_property(property_name, change)
            last.snapshot = deep_clone(this)
            return
        }

        history.add(
            TraceEntry(
                timestamp = Clock.utc_now(),
                caller_class = caller.class,
                caller_method = caller.method,
                changes = [change],
                snapshot = deep_clone(this)
            )
        )
    }

    function mutation_history(): List[TraceEntry] {
        return history
    }
}
```

The key design point:

One caller method should produce one mutation slice.

Not one audit record per setter.

---

## State Provenance

Every significant mutation should answer:

* What changed?
* Who changed it?
* Why was it changed?
* When did it happen?

Example:

```pseudo
Mutation {
    operation: ApplyDiscount
    timestamp: 2026-05-01T10:15:00Z

    changes:
        total:
            100 → 80
}
```

The goal is explainability, not auditing for compliance.

---

## Why It Matters

When investigating failures, AI agents spend enormous effort reconstructing state history.

Typical workflow:

```text
Current State
    ↓
Search Logs
    ↓
Search Events
    ↓
Search Handlers
    ↓
Search Repositories
    ↓
Guess Root Cause
```

With ADTOs:

```text
Current State
    ↓
Inspect Mutation History
    ↓
Identify Root Cause
```

The debugging path becomes dramatically shorter.

---

## Relationship to AI Oriented Architecture

AI Oriented Architecture favors local reasoning.

An agent should not need to reconstruct large portions of the system to understand why an object reached a particular state.

ADTOs make state transitions locally observable.

Instead of searching the ecosystem, the explanation travels with the object.

---

## What This Is Not

ADTOs do not mean:

* storing every setter call;
* creating massive object histories;
* replacing logs;
* replacing tracing systems;
* replacing event streams.

The objective is not perfect historical reconstruction.

The objective is making important state transitions self-explanatory.

---

## Implementation Approaches

An ADTO may be implemented using:

* mutation journals;
* embedded provenance metadata;
* event-derived state histories;
* lightweight state-transition records;
* domain-event aggregation.

The implementation is secondary.

The architectural goal is explainable state.

---

## Recommendations

For AI engineering teams:

* Treat DTOs as contracts.
* Treat ADTOs as runtime state carriers.
* Record business-significant mutations.
* Prefer mutation slices over field-level noise.
* Capture why a state changed, not just what changed.
* Expose mutation history to debugging tools and AI agents.
* Make important state transitions self-describing.

Stop forcing agents to reconstruct object history from scattered infrastructure artifacts.

Let state explain itself.
