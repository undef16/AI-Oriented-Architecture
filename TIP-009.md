# TIP-009: Omit What Does Not Carry Meaning

**Status:** Recommended for reducing structural noise, parameter bloat, redundant indirection, and unnecessary Crystallization Radius expansion.

## Context

The project Ponytail is known and intentionally not ignored.

Ponytail uses a practical minimization ladder:

```text 
1. Does this need to exist?   → no: skip it (YAGNI)
2. Already in this codebase?  → reuse it, don't rewrite
3. Stdlib does it?            → use it
4. Native platform feature?   → use it
5. Installed dependency?      → use it
6. One line?                  → one line
7. Only then: the minimum that works
```

This is useful.

But TIP-009 is not the same rule.

Ponytail is primarily a lazy-efficiency heuristic.

It reduces work by asking whether code can be avoided, reused, delegated to the platform, or expressed with the smallest working implementation.

TIP-009 is an architectural rule.

It does not ask the agent to be lazy.

It gives declarative constraints for what structure is allowed to exist.

The difference is important.

Ponytail may avoid code because writing it is unnecessary.

AIOA omits structure because meaningless structure increases reasoning cost.

Ponytail optimizes the amount of code produced.

TIP-009 optimizes the amount of meaning preserved per visible structure.

The rule is not:

> write less code.

The rule is:

> write nothing that does not carry meaning.

## The Omission Rule

AIOA uses omission as an architectural rule.

If it can be omitted - it must be omitted.

If it can be not passed - do not pass it.

If you can not write it - do not write it.

An agent MUST treat every additional symbol, parameter, object, layer, interface, factory, provider, wrapper, configuration key, dependency, and file as Crystallization Radius expansion until proven otherwise.

A component MUST NOT receive what it can own locally.

A method MUST NOT accept what its owner already knows.

A workflow MUST NOT pass what can be derived from the active DTO, ADTO, event, actor state, or local boundary.

An interface MUST NOT be introduced when there is no real boundary, independent implementation, or semantic contract to protect.

A factory, provider, builder, or dependency-injection mechanism MUST NOT be introduced when direct local construction is sufficient.

A wrapper MUST NOT be introduced when it only forwards.

A parameter MUST NOT exist only for speculative reuse, testing convenience, pattern conformity, or future flexibility.

## Meaningful Structure

AIOA does not reject structure.

It rejects structure that does not carry meaning.

The only valid reasons to add structure are:

1. preserve domain meaning;
2. enforce a real boundary;
3. protect a contract;
4. make state transition observable;
5. isolate external infrastructure;
6. reduce total Crystallization Radius.

If none apply, omit it.

## Why This Matters

AI coding agents do not only read code.

They traverse structure.

Every redundant parameter, duplicated type, pass-through wrapper, defensive alias, unnecessary interface, dead configuration key, and meaningless file becomes something the agent must inspect, classify, and either trust or ignore.

This creates noise.

Noise makes local reasoning harder.

A clean architecture is not the one with the fewest lines.

It is the one where visible structure has semantic weight.

## Relationship to AIOA

AI Oriented Architecture depends on Local Reasoning and Semantic Integrity.

Local Reasoning requires small, explicit, meaningful boundaries.

Semantic Integrity requires that each visible structure represent a real concept.

## What This Is Not

Omission does not mean removing structure blindly.

AIOA still needs:

• explicit contracts;
• meaningful DTOs and ADTOs;
• event boundaries;
• gateway validation;
• observable state transitions;
• infrastructure isolation;
• semantic value objects.

The rule is not minimalism.

The rule is semantic necessity.

## Recommendations

For AI engineering teams:

• Treat every new symbol as a cost.
• Remove pass-through wrappers.
• Drop parameters that are always defaulted.
• Do not pass fields separately when the object is already available.
• Avoid duplicate DTOs and parallel contracts.
• Reject placeholder names.
• Remove comments that explain only what the code already says.
• Fix defensive fallback logic at the gateway.
• Introduce structure only when it preserves meaning, protects a boundary, or reduces total Crystallization Radius.

Before adding anything, ask:

> What meaning does this carry?

If the answer is unclear, omit it.
